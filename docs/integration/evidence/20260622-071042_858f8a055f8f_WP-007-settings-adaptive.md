## Scope

This evidence records a bounded Settings-window visual review for effective
minimum window size plus Dark/Light adaptive appearance under Increased
Contrast, Reduce Transparency, and Reduce Motion. It is a WP-007 increment, not
full visual/accessibility signoff.

code head `b792efededfd`. Since then, the branch has only added docs/evidence;
no app source changed before this screenshot pass.

## Procedure

The installed app was launched locally and the Settings window opened through
the native menu/keyboard path. Chrome/browser tooling was not used.

The initial global state was Dark Mode, with no saved values for
`AppleIncreaseContrast`, `AppleReduceTransparency`, or `AppleReduceMotion`.

Adaptive capture sequence:

```bash
defaults write -g AppleIncreaseContrast -bool true
defaults write -g AppleReduceTransparency -bool true
defaults write -g AppleReduceMotion -bool true
```

Light adaptive capture additionally used:

```applescript
tell application "System Events" to tell appearance preferences to set dark mode to false
```

The system was restored after capture:

```applescript
tell application "System Events" to tell appearance preferences to set dark mode to true
```

```bash
defaults delete -g AppleIncreaseContrast
defaults delete -g AppleReduceTransparency
defaults delete -g AppleReduceMotion
```

Post-restore check showed `AppleInterfaceStyle` back to `Dark`, with the three
adaptive defaults unset.

## Review Findings

Independent macOS design critique reviewed the screenshot set and found no
release-blocking Settings UI defect:

- the Settings surface remains native SwiftUI/macOS in both Light and Dark;
- no fake glass, blur dependency, material mismatch, or Reduce Motion/Reduce
  Transparency regression was visible;
- the effective-minimum screenshots do not prove a narrower layout beyond the
  native minimum, so the limitation is documented here;
- focused text-field highlights in the path/model fields are expected macOS
  focus/selection behavior and are treated as capture artifacts, not release
  blockers;
- secondary explanatory text is muted but legible in the captures.

Source inspection found the relevant Settings explanatory text uses semantic
SwiftUI `.secondary` foreground styling and design-system semantic status
colors rather than custom fixed gray foreground colors.

## Interpretation

This evidence supports the Settings window for:

- Dark default appearance;
- effective native minimum size at `900x648` logical points;
- Dark appearance with Increased Contrast, Reduce Transparency, and Reduce
  Motion enabled;
- Light appearance with Increased Contrast, Reduce Transparency, and Reduce
  Motion enabled;
- normal visible focus/selection behavior in Settings fields;
- lower Settings content scrolling into view.

No code changes were required by this pass.

## Limitations

- Increased Text Size was not captured.
- Inactive-window appearance was not captured.
- This is screenshot/design-review evidence, not full keyboard navigation or
  VoiceOver traversal.
- The screenshot set focuses on the Workspace pane. Providers and Legacy panes
  already have normal Light/Dark evidence from `EV-20260622-015`, but they were
  not re-captured under the adaptive settings in this pass.
