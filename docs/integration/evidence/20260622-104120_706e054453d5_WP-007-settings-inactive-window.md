## Procedure

The installed native app was launched locally. Browser tooling and provider
credentials were not used.

```bash
defaults write local.paperbanana.gui settings.repoPath -string "/Users/jeff/Codex_projects/PaperBanana"
open -n -b local.paperbanana.gui
osascript -e 'tell application "PaperBanana" to activate' \
  -e 'tell application "System Events" to tell process "PaperBanana" to keystroke "," using command down'
```

The native Settings window opened as `Workspace`. Its CoreGraphics window id and
bounds were:

```text
17079    Workspace    { Height = 648; Width = 900; X = 40; Y = 80; }
```

Screenshots were captured with:

```bash
screencapture -x -o -l 17079 docs/integration/evidence/screenshots/20260622-settings-inactive/settings-*.png
```

Inactive-window captures were taken after:

```applescript
tell application "Finder" to activate
```

Dark and Light appearance were set explicitly for the paired captures:

```applescript
tell application "System Events" to tell appearance preferences to set dark mode to true
tell application "System Events" to tell appearance preferences to set dark mode to false
```

The original appearance mode was restored after capture.

## Review Findings

- The Settings scene remains native and legible in Dark active, Dark inactive,
  Light active, and Light inactive states.
- Inactive-window captures correctly mute the traffic-light controls and selected
  Settings tab accent while preserving the readable hierarchy.
- The Workspace pane keeps its form content visible at the native Settings
  window size with no overlap, clipping, or card-within-card regression in the
  captured viewport.
- Status colors remain distinguishable in both appearances. The warning pill,
  green readiness rows, and secondary explanatory text remain readable in the
  inactive-window captures.
- The path-field selection highlight visible in the captures is consistent with
  prior Settings evidence and does not obscure adjacent labels or subsequent
  content.
